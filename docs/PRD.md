# PRD：Apple ID 代充业务管理系统 + 兑换码自动发货系统

## 1. 项目目标

开发一个后台管理系统，用于管理两条独立业务线：

1. Apple ID 代充业务管理系统
2. 兑换码自动发货系统

系统要支持客户管理、来源平台管理、订单管理、Apple ID 管理、余额成本核算、续费任务、兑换码库存、淘宝/闲鱼自动发货、权限管理、操作日志和报表统计。

## 2. 总体业务边界

### 2.1 Apple ID 业务区

Apple ID 业务区用于代充和订阅开通业务。

核心能力：

- 批量导入 Apple ID
- 管理 Apple ID 账号、密码、手机号、密保、地区、余额、状态
- 管理 Apple ID 余额充值
- 使用移动加权平均成本计算余额成本
- 录入客户订单
- 选择业务后自动匹配可用 Apple ID
- 按锁定规则判断 Apple ID 是否可用
- 生成开通记录
- 生成余额消费记录
- 计算订单利润
- 到期前生成提醒
- 联系客户确认是否续费
- 客户续费时生成充值/等待自动续费任务
- 客户不续费时生成取消订阅任务
- 同一个 Apple ID 多个业务快到期时，生成 Apple ID 操作计划
- 管理 Apple ID 自动化任务
- 生成 Apple ID 报表

### 2.2 兑换码业务区

兑换码业务区用于管理某个网站/产品的兑换码库存和自动发货。

核心能力：

- 设置兑换码业务
- 批量导入兑换码
- 按面值管理库存
- 淘宝订单同步/导入
- 闲鱼订单同步/导入
- 商品/SKU 映射到兑换码面值
- 客户下单后按面值自动锁定兑换码
- 自动或半自动发货
- 防止重复发码
- 发货失败转人工
- 售后补发
- 退款/售后记录
- 兑换码利润报表

### 2.3 两个业务区必须分开

Apple ID 业务区和兑换码业务区没有业务依赖关系。

- Apple ID 订单不使用兑换码库存。
- 兑换码订单不使用 Apple ID。
- Apple ID 报表和兑换码报表分开。
- Apple ID 成本和兑换码成本分开。
- Apple ID 续费任务和兑换码发货任务分开。

## 7. Apple ID 锁定规则

### 7.1 规则1：按业务锁定

默认规则。

定义：

同一个 Apple ID，在同一个业务未到期期间，不能重复使用。

示例：

- A 账号已经开了 GPT，GPT 未到期，不能再开 GPT。
- A 账号已经开了 GPT，但可以开 Gemini。
- A 账号 GPT 已到期，可以再次开 GPT。

优点：

- 成本低
- 账号利用率高
- 适合批量代充

### 7.2 规则2：全局锁定

可选规则。

定义：

同一个 Apple ID，只要有任何未到期业务，就不能再开其他业务。

示例：

- A 账号 GPT 未到期，不能开任何新业务。
- A 账号所有业务都到期后，才可以开新业务。

优点：

- 最清晰
- 风险低
- 适合高价值客户或特殊业务

缺点：

- 成本高
- 账号利用率低

## 13. Apple ID 操作计划

### 13.1 场景

如果同一个 Apple ID 里面有多个业务快到期，系统要按 Apple ID 分组生成操作计划。

示例：

- GPT：客户不续费，需要取消订阅
- Gemini：客户续费，需要充值等待自动续费
- Claude：客户未回复，需要继续询问

系统要生成一个 Apple ID 操作计划，而不是三个零散提醒。

### 13.2 操作计划字段

- 当前余额
- 当前平均成本

- 预计续费金额
- 建议充值金额
- 是否存在误扣费风险
- 操作状态
- 总备注

### 13.3 操作计划明细

每条明细对应一个开通记录。

字段：

- 开通记录
- 客户
- 业务
- 当前套餐
- 目标套餐
- 到期时间
- 客户决定
- 动作类型
- 预计扣费

### 13.4 风险提醒

系统自动识别：

- 客户不续费但订阅未取消
- 同一 Apple ID 有多个即将扣费业务
- 余额足够但有不续费业务未取消
- 客户要续费但余额不足
- 客户说续费但未付款
- 已充值但其他不续费业务未取消
- 到期当天未处理
- 自动扣费失败
- 客户未回复
- 任务过期未完成

### 17.1 边界

兑换码系统只处理：

发货模板规则：

- 发货模板只用于兑换码业务的淘宝/闲鱼自动回复。
- 模板固定为 `type=delivery`、`channel=customer_service`。
- 系统通知模板走通知设置，不允许绑定到兑换码发货流程。
- 缺货处理

导入字段：

- 兑换码
- 面值

- 过期时间

导入时：

- 检查空码
- 检查重复码
- 按面值分类
- 生成批次
- 统计数量
- 错误行导出

### 17.4 兑换码状态

- unsold：未售
- locked：已锁定
- delivered：已发送
- delivery_failed：发货失败
- after_sale：售后中
- reissued：已补发
- voided：已作废
- refunded：已退款

### 17.5 自动发货流程

1. 客户下单
2. 系统同步淘宝/闲鱼订单
3. 根据商品 ID/SKU/关键词识别面值
4. 检查订单是否已付款
5. 检查订单是否已发货
6. 锁定对应面值兑换码
7. 生成发货内容
8. 调用平台发货接口或进入半自动发货
9. 发货成功后标记兑换码已发送
10. 生成发货记录
11. 计算利润

### 17.6 面值匹配规则

支持精确面值：

- 买 100，只发 100 面值兑换码
- 买 200，只发 200 面值兑换码

支持组合发货：

- 买 300，可以发 3 个 100
- 买 500，可以发 200 + 300

默认建议精确面值。

### 17.7 防重复发码规则

- 同一个兑换码只能发给一个订单
- 同一个平台订单不能重复发货
- 已退款订单不能自动发货
- 已发货订单不能再次自动发货
- 补发必须走售后补发流程

状态流转：

- 未售 → 已锁定 → 已发送
- 已锁定 → 发货失败 → 释放回未售
- 已锁定 → 发货失败 → 转人工

## 24. Apple ID 充值礼品卡代码管理补充

Apple ID 充值记录需要支持录入礼品卡代码或充值代码，用于追踪每次充值来源和防重复使用。

### 24.1 字段和边界

- 礼品卡代码完整内容必须加密保存。
- 保存 `gift_card_code_hash`，用于检测同一礼品卡代码是否重复录入。
- 保存 `gift_card_code_tail`，用于列表和详情页默认显示后 4 位。
- 礼品卡代码只属于 Apple ID 充值记录。
- 礼品卡代码不得进入兑换码库存。
- Apple ID 充值礼品卡代码不得用于兑换码自动发货。
- 兑换码库存里的兑换码不得作为 Apple ID 充值记录的礼品卡代码来源。

### 24.2 录入规则

录入 Apple ID 充值时，如果填写礼品卡代码：

1. 系统先计算 `gift_card_code_hash`。
2. 检查是否已存在相同 hash。
3. 如果重复，阻止保存并提示礼品卡代码已使用。
4. 保存完整代码的加密密文。
5. 保存后 4 位到 `gift_card_code_tail`。
6. 充值金额、本次平均成本、人民币成本仍按原移动加权平均成本规则处理。

### 24.3 查看规则

默认页面只显示礼品卡代码后 4 位。

查看完整礼品卡代码必须满足：

- 用户拥有 `apple.topup.gift_code.view_full` 权限。
- 后端重新校验权限，不能只靠前端隐藏按钮。
- 查看动作必须写审计日志。
- 返回结果不得被前端或中间层公共缓存。

### 24.4 审计要求

以下操作必须写入审计日志：

- 新增带礼品卡代码的 Apple ID 充值记录。
- 查看完整礼品卡代码。
- 导出包含礼品卡代码信息的充值记录。
- 修改或作废包含礼品卡代码的充值记录。
