# 开发到上线执行路线图

## 1. 文档定位

本文档用于指导本项目从当前开发状态推进到正式上线。

它不是 PRD，也不是接口文档，而是上线前的执行清单：哪些模块已经具备第一版能力，哪些只是占位，哪些必须在上线前补齐，哪些可以作为上线后的增强迭代。

当前项目路径：

```text
/Users/wangchao/Desktop/源码文件夹/ai软件代充管理软件
```

当前技术栈：

- 后端：NestJS + TypeScript
- 前端：Vue 3 + Element Plus + Vite
- 数据库：PostgreSQL
- ORM：Prisma
- 队列：Redis + BullMQ
- 部署：Docker Compose

## 2. 当前上线判断

当前系统已经完成后台基础工程和主要业务模块的第一版实现，本地等效上线验收已经通过，但还不能直接按“全自动发货系统”标准上线。

可以进入业务验收的能力：

- 登录、用户、角色、权限、操作日志
- 客户、来源平台、附件中心
- Apple ID 管理、敏感字段加密和审计
- Apple ID 充值、消费、余额修正、移动加权平均成本
- Apple ID 订单录入、自动匹配、锁定规则、开通记录、利润报表
- Apple ID 续费任务中心、待取消订阅、待充值续费、等待自动续费、操作计划
- 兑换码业务设置、发货模板、批量导入、去重、库存、半自动发货、售后补发、利润报表
- 通知中心、Telegram 配置、站内通知、通知规则和通知日志
- 安全中心、数据中心、运维监控、网站维护、审计日志中心、平台接口状态
- 生产部署底座、健康检查、生产环境变量模板、备份恢复脚本、上线烟测脚本

当前 P0 剩余人工和生产环境门禁：

- 确认是否创建首个 Git commit 并推送远程仓库
- 使用 `npm run prod:env:init` 生成生产 `.env.production` 强随机密钥骨架，填入真实域名后运行 `npm run prod:env:check`
- 每轮上线前使用 `npm run launch:status` 复核当前未完成项、生产环境变量状态和 Git 状态
- 在预发布或生产等效环境再次执行 `npm run acceptance:launch`
- 配置生产 HTTPS、域名、外部备份存储、远程日志采集和回滚方案

Telegram Bot Token / Chat ID 填写位置已保留在通知中心；`FIRST_RELEASE_MODE=semi_auto` 时可以首发后补，再完成真实测试发送。

可以后置到上线后增强，或在“必须全自动上线”策略下提前处理的能力：

- 淘宝、闲鱼真实开放平台授权、签名、订单同步、发货、退款回调
- Apple ID 真实自动化 Worker
- 数据清理、重复合并等高风险数据操作的真实执行器

## 4. 开发到上线阶段

### Stage 0：当前基线冻结

目标：确认当前代码、文档、数据库和环境可以作为后续开发基线。

执行项：

- 运行 `npm run doctor`
- 运行 `npm run launch:status`
- 运行 `npm run check`
- 运行 `npm run prod:config:example`
- 确认 Docker 本地基础设施可启动
- 确认 `/api/health/live` 和 `/api/health/ready` 正常
- 确认是否需要创建初始 Git commit 和推送远程仓库

出口标准：

- 基础命令全部通过
- 当前文档和代码状态明确
- 未提交文件状态已被用户接受，或已按用户要求建立 Git 基线

### Stage 1：P0 上线缺口修复

目标：修复会影响上线稳定性、安全性和用户验收的硬缺口。

执行项：

- 上线检查清单页面从占位改为真实验收任务页面
- 403、404、系统维护模式页确认可访问且体验正常
- 维护模式接入后台全局拦截，避免误操作期间继续访问核心页面
- 生产环境变量校验覆盖所有真实必填项和占位密钥拦截
- 权限种子和菜单权限逐项核对
- 敏感字段查看流程回归：权限、原因、审计、敏感访问日志
- 附件上传、下载、审计、生产存储路径回归
- 数据库备份脚本在本地或预发布环境演练一次
- 上线烟测脚本覆盖 Admin、API live、API ready

出口标准：

- 不存在明显上线阻塞项
- 敏感信息不能越权查看
- 维护模式和异常页面不会造成死路由或空白页
- 备份和恢复流程有可验证记录

### Stage 2：核心业务端到端验收

目标：用接近真实业务的数据验证两条业务线都能跑通。

Apple ID 业务验收：

- 创建客户和来源平台
- 创建 Apple ID 账号
- 录入 Apple ID 充值，验证礼品卡代码加密、hash 去重、尾号展示
- 录入消费和余额修正，验证移动加权平均成本
- 创建 Apple ID 业务设置和平台映射
- 录入订单，验证自动匹配、锁定规则、开通记录、扣费、利润
- 生成续费任务，验证待取消、待充值、等待自动续费和操作计划
- 查看敏感字段，验证权限和日志
- 导出或查看报表，验证金额口径

兑换码业务验收：

- 创建兑换码业务设置和平台映射
- 批量导入兑换码，验证去重、加密、尾号、库存状态
- 手工导入订单，验证自动匹配面值和锁码
- 生成发货内容，验证防重复发货
- 创建售后补发，验证新码锁定、损耗成本、日志
- 查看利润报表，验证订单、成本、退款、利润

出口标准：

- 两条业务线都能在后台完成闭环
- Apple ID 和兑换码数据、订单、成本、报表没有混用
- 每个高风险动作都有审计记录

### Stage 3：平台接口和自动化策略确认

目标：决定第一版上线是“半自动可运营”，还是必须“全自动发货”。

如果第一版允许半自动上线：

- 淘宝/闲鱼真实接口可以后置
- 平台订单先通过手工导入或人工同步处理
- 自动发货失败统一转人工
- Apple ID 自动化任务保持人工验证兜底

如果第一版必须全自动上线：

- 接入淘宝 OAuth、签名、订单同步、发货、退款同步
- 接入闲鱼授权、订单同步、电子凭证或无需物流发货
- 启用已接入的平台轮询 Worker，真实 Webhook 后续按平台能力补充
- 接入真实 Apple ID 自动化 Worker
- 对平台接口失败、授权过期、发货异常增加重试和通知

出口标准：

- 上线策略明确
- 如果走半自动，页面文案和权限不能伪装成全自动
- 如果走全自动，平台真实接口必须完成联调和失败兜底

### Stage 4：安全、数据和运维加固

目标：让系统具备生产运行的最低安全和恢复能力。

执行项：

- 生产 JWT、字段加密、hash、MinIO、数据库密码全部使用强随机值
- 确认 `.env.production` 不提交 Git
- 批量导出、敏感查看、角色权限变更写日志
- 登录失败和异常登录通知可触发
- 如本轮启用 Telegram，则真实测试发送通过；半自动首发可后补
- 数据库备份成功并记录文件位置
- 恢复脚本在非生产环境演练通过
- 运维监控能展示 API、数据库、Redis、队列和磁盘状态
- 关键错误写入 error_logs 或平台接口日志

出口标准：

- 生产密钥不使用占位值
- 有可恢复的数据库备份
- 管理员能看到故障、通知和审计证据

### Stage 5：预发布环境部署

目标：在接近生产的环境中完整演练部署和业务流程。

执行项：

- 准备预发布 `.env.production`
- 构建 Docker 镜像
- 启动 `docker-compose.prod.yml`
- 执行 Prisma migration
- 执行 seed
- 执行 `BASE_URL=<staging_url> npm run prod:smoke`
- 导入验收数据
- 完整跑一遍 Stage 2 业务验收
- 记录所有问题并修复

出口标准：

- 预发布环境稳定运行
- 健康检查和业务验收通过
- 不再出现 P0 级别缺陷

### Stage 6：正式上线

目标：把稳定版本发布到生产环境。

执行项：

- 冻结上线版本
- 上线前备份数据库
- 确认 DNS、HTTPS、Nginx 反代
- 构建并启动生产容器
- 执行数据库迁移
- 执行 seed 或确认权限种子已存在
- 运行生产烟测
- 管理员登录验证
- 验证核心业务入口
- 验证通知、备份、日志、平台状态
- 开放给业务人员使用

出口标准：

- 生产域名可访问
- 管理员可登录
- API ready 正常
- 核心业务流程可操作
- 有备份和回滚方案

### Stage 7：上线后观察和迭代

目标：上线后先稳定，再扩展。

前 24 小时观察：

- 登录失败率
- API 错误率
- 数据库和 Redis 状态
- 队列积压
- 自动发货失败
- Apple ID 续费任务异常
- Telegram 通知失败
- 磁盘空间
- 备份结果

上线后优先迭代：

- 淘宝/闲鱼真实自动发货
- Apple ID 真实自动化 Worker
- 远程日志采集和异地备份存储
- 更多列表页统一表格迁移
- 风控中心和客服工单

`npm run acceptance:launch` 是本地等效上线验收总入口，已串行包含 `npm run check`、生产 Compose 示例校验、业务闭环验收、安全验收、数据库备份、临时库恢复演练、手工门禁只读核验和 Git 提交前安全检查。本地验收会把缺少真实生产域名和上线检查清单手工项视为外部门禁提示；`FIRST_RELEASE_MODE=semi_auto` 时 Telegram 可后补。预发布或生产等效验收必须使用 `REQUIRE_PROD_ENV=1 npm run acceptance:launch` 强制校验生产环境变量和手工门禁。

`npm run launch:status` 是上线前只读状态速览入口。它会按 `FIRST_RELEASE_MODE` 区分“首版阻塞项”和“上线后自动化路线图”：`semi_auto` 模式下，淘宝/闲鱼真实 Adapter 和 Apple ID 真实 Worker 不阻断首个内部可运营版本；`full_auto` 模式下，Phase 17 未完成项全部视为上线前阻塞项。

上线策略不变时，先运行 `npm run prod:env:set-mode -- --dry-run --mode=semi_auto` 验证，再去掉 `--dry-run` 写入 `.env.production`。该脚本只更新 `FIRST_RELEASE_MODE`，不会打印或重置生产密钥。

`npm run launch:gates` 是上线前手工门禁只读核验入口，会同时检查 `.env.production` 和上线检查清单中的 `prod_env`、`git_baseline` 状态；`FIRST_RELEASE_MODE=semi_auto` 时 Telegram 显示为 deferred，可后补。该命令不会发送 Telegram，不会写数据库，不会输出明文 token；预发布或生产等效环境可使用 `npm run launch:gates:strict` 让未通过项返回失败退出码，也可以设置 `REQUIRE_MANUAL_GATES=1 npm run acceptance:launch`。

`npm run launch:checklist` 是上线检查清单手工证据记录入口，只允许更新 `telegram_test`、`prod_env`、`git_baseline` 三个手工项。真实 Telegram 测试、真实生产域名校验或 Git 基线确认完成后，用该命令写入状态和证据；它会拒绝常见密钥格式出现在证据文本里，并写入审计日志。`telegram_test` 在半自动首发下可以保持 pending，后续填入并测试成功后再记录 `passed`；写入 `passed` 前仍会检查真实状态。生产 env 和 Git 基线门禁通过后再运行 `npm run release:ready` 和 `REQUIRE_PROD_ENV=1 REQUIRE_MANUAL_GATES=1 npm run acceptance:launch`。示例：`npm run launch:checklist -- --id=git_baseline --status=passed --evidence="initial commit pushed to origin/main; working tree clean"`。

拿到真实生产域名后，先运行 `npm run prod:env:set-domain -- --dry-run --app-url=https://your-domain.com` 验证，再去掉 `--dry-run` 更新 `.env.production`。该脚本只改 `APP_PUBLIC_URL` 和 `CORS_ORIGIN`，不会打印或重置生产密钥。

提交或推送前必须运行 `npm run git:readiness`。该检查会拦截 `.env`、`.env.production`、备份、上传文件、构建产物、Telegram Bot Token 和常见云厂商/API 密钥进入 Git 候选文件。

首次提交前建议额外运行 `npm run git:readiness:verbose`，查看候选文件按顶层目录和扩展名的摘要，以及前 40 个候选文件样本。

生产或预发布部署后运行：

```bash
BASE_URL=http://localhost:8080 npm run prod:smoke
```

接入域名后运行：

```bash
BASE_URL=https://example.com npm run prod:smoke
```

## 7. 下一步执行顺序

从当前状态继续开发，建议严格按下面顺序：

1. 先确认是否允许创建首个 Git commit 并推送到远程仓库。
2. 准备生产 `.env.production`，运行 `npm run prod:env:check`。
3. 决定第一版是否允许半自动上线。
4. 如果允许半自动上线，进入预发布部署和验收；Telegram 留在通知中心后补。
5. 如果必须全自动上线，先暂停上线准备，优先接淘宝、闲鱼、Apple ID Worker 和 Telegram 真实测试。
6. 预发布环境完整演练通过后，再进入正式上线。
